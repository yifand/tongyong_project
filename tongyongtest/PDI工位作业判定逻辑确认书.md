PDI工位作业判定逻辑确认书
一、背景与目的
根据2026年3月2日内部会议与客户沟通的最终共识，对PDI作业的判定逻辑进行根本性调整：不再采用单一的“头帧/尾帧”计时或周期性抽帧统计，而是由边缘计算盒子提供三个基础状态的实时输出，业务平台（应用层）基于这些状态的时间序列组合，灵活定义各种业务规则（如作业开始、结束、合规判定等）。此方案可覆盖检测人员长时间在车内作业（占比超50%）的场景，避免误判，同时具备良好的扩展性和可配置性。
本确认书旨在锁定底层状态定义、八种可能组合、盒子与业务平台的分工，以及后续开发与验收的依据。

三、底层状态定义（算法侧输出）
边缘计算盒子对每个工位实时输出以下三个独立的状态（均为布尔值）：
door_open：车门是否开启
true：车门处于打开状态
false：车门处于关闭状态
person_present：工位区域内是否检测到人员（包括头肩、人体）
true：区域内有人
false：区域内无人
person_entering_exiting：人员是否正处于进出车门的瞬间（即身体部分在车内、部分在车外）
true：正在进出
false：未处于进出状态（包括完全在车外或完全在车内）
说明：进出状态与人员状态可能同时为真（例如人正在钻入时，部分身体仍在区域外，但算法仍可检测到进出动作）。该状态独立存在，用于精确捕捉人员进出车内的动作。

五、判定逻辑（应用层实现）
业务平台基于接收到的实时状态流，通过时间序列组合来判定作业的各个阶段。具体规则由应用层根据客户需求灵活定义，以下为典型示例：
5.1 人员进入车内判定
当出现以下连续状态序列时，可判定人员已进入车内：
首先检测到 状态7（door_open=true, person_present=true, entering_exiting=false）→ 人员站在车外，车门开启；
随后检测到 状态8（door_open=true, person_present=true, entering_exiting=true）→ 人员开始进入；
然后检测到 状态3（door_open=false, person_present=true, entering_exiting=false）→ 车门关闭，区域内有人（人员已在车内）。
此时记录“人员进入车内”事件，并开始累积车内作业时间。
5.2 人员离开车内判定
类似地，当出现：
状态3（车门关，有人，未进出）→ 人员在车内；
状态8（车门开，有人，进出）→ 人员开始出来；
状态7（车门开，有人，未进出）→ 人员已出车外；
状态5（车门开，无人，未进出）或状态1（车门关，无人）→ 人员离开工位。
可判定人员离开。
5.3 工位作业时长计算
作业开始时刻：可定义为第一次出现状态7（车门开启且有人）或状态8（进出）的时刻。
作业结束时刻：可定义为最后一次出现人员状态为真的时刻（如状态1或5持续超过阈值）。
总时长 = 结束时刻 - 开始时刻。
5.4 合规判定
应用层可根据工艺要求（如左前门作业需≥12分钟）比较实际时长，若不足则触发报警。
5.5 多车门合并流程
对于左前+左后合并流程（总时长15分钟），应用层可分别跟踪两个车门的作业状态，以第一个车门开启为开始，最后一个车门作业完成为结束，计算总时长。
5.6 滑移门识别
针对面包车，算法同样输出上述三个状态，滑移门的进出状态识别需专门训练。
六、业务平台与边缘计算盒子协作分工
6.1 边缘计算盒子（现场端）职责
实时接入视频流，运行三个独立算法（车门开关检测、人员存在检测、进出状态检测）。
对每个工位，以固定频率（如每秒1次或每帧）输出基础状态三元组，通过接口推送给业务平台。
同时提供抓拍截图（如关键事件时刻）和日志。
推送数据格式（参考接口文档4.11.2）
json
{
"box_id": "设备ID",
"timestamp": 1646123456,
"channel_id": "通道ID",
"states": {
"door_open": true,
"person_present": true,
"person_entering_exiting": false
},
"snapshot": {   // 可选，关键帧截图
"target_image": "base64...",
"scene_image": "base64..."
}
}
可通过上传中心配置（通用类型）将数据推送到业务平台指定的HTTP端点。
6.2 业务平台（金桥端）职责
接收并存储实时状态流，建立时间序列数据库。
实现业务规则引擎，根据状态序列判定作业开始、结束、违规等事件。
提供前端界面（瀑布流、红绿灯、报警弹窗、报表）。
支持手机端推送（如微信、钉钉）。
提供日志查询和追溯功能。
七、数据采集与标注要求
为保证算法准确性，需采集真实作业场景视频，覆盖以下内容：
常规场景：左前门、左后门、滑移门在不同光照、角度下的完整作业过程（开门→人员进出→关门→离开）。
进出状态专项：重点采集人员进出车门的瞬间视频，标注进出状态（是/否）。
车内作业场景：采集人员进入车内并关闭车门后的视频（用于训练人员存在推断）。
滑移门专项：面包车滑移门开启/关闭及进出视频，至少500段，由客户提供。
标注要求：每段视频需逐帧或按关键帧标注三个状态的真值。
八、双方确认
我方（算法提供方）将严格按照上述定义开发三个基础状态算法，并按照接口规范实时推送数据。
贵方（客户方）确认该设计满足业务需求，并负责应用层规则的定义与实现。

| 车门状态    | 算法输出的二值状态：车门开启（open）或关闭（closed）。                                                                  |
| 人员状态    | 算法输出的二值状态：工位区域内有人（person present）或无人（person absent）。                                              |
| 进出状态    | 算法输出的二值状态：人员是否正处于进出车门的瞬间（如头钻入车内、身体卡在车门处等），是（entering/exiting）或否（not）。该状态独立于人员状态，用于判断人员进入或离开车内的动作。 |
| 基础状态三元组 | 每一时刻（每帧或每固定间隔）由算法输出的三个二值状态组合，共8种可能。                                                               |
| 业务规则    | 应用层根据时间序列上的基础状态组合，定义作业开始、结束、合规判定等逻辑。                                                              |
