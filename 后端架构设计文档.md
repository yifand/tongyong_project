## 1. 架构目标与约束

### 1.1 架构目标

- **单服务部署**: 采用单体架构，降低运维复杂度，适配当前团队规模与业务量。
- **数据隔离**: 通过 `site_id` 字段实现金桥库、凯迪库的数据级隔离。
- **实时处理**: 状态流数据接收与规则引擎处理延迟 ≤ 3秒。
- **规则可配**: 状态机转换规则、报警阈值、PDI标准工时均支持配置化。
- **算法预留**: 独立设计算法数据入口模块，便于未来对接更多边缘算法或算法中台。

### 1.2 架构约束

| 约束项 | 说明 |
|--------|------|
| 单体服务 | 不允许拆分微服务，所有业务模块运行在同一个JVM进程中 |
| Java 17 | 必须基于Java 17 LTS版本开发 |
| PostgreSQL | 唯一持久化关系型数据库，禁止引入其他OLTP数据库 |
| 算法数据入口 | 必须预留HTTP/MQTT数据接收接口，与业务服务解耦 |
| 单元测试 | 核心业务层代码覆盖率 ≥ 70% |

## 5. 包结构规范

### 5.1 统一包结构

每个业务模块均采用以下分层包结构：

```
com.vdc.pdi.<module-name>/
├── controller/           # 控制器层：REST API入口
├── service/              # 应用服务层：业务编排
│   └── impl/
├── domain/               # 领域层
│   ├── entity/           # JPA实体
│   ├── repository/       # Repository接口（JPA/QueryDSL）
│   ├── event/            # 领域事件
│   └── vo/               # 值对象
├── dto/                  # 数据传输对象
│   ├── request/
│   └── response/
├── mapper/               # MapStruct映射器
├── config/               # 模块级配置
└── exception/            # 模块级异常（可选）
```

### 5.2 跨模块调用规范

- **上层可调用下层，禁止反向依赖**：`controller` → `service` → `domain`
- **同层不直接调用**：`controller` 之间禁止直接调用，需通过 `service` 层
- **模块间调用**：A模块的 `service` 可注入B模块的 `service` 接口，禁止直接引用 `controller` 或内部实现类
- **DTO隔离**：模块间通过 `dto` 交互，禁止直接暴露 `entity`

## 6. 数据库设计原则

### 6.1 命名规范

- 表名：小写字母 + 下划线，如 `pdi_task`, `alarm_record`, `state_stream`
- 字段名：小写字母 + 下划线，如 `site_id`, `channel_id`, `created_at`
- 主键：统一使用 `id`，类型 `BIGINT`，自增
- 时间字段：`created_at`, `updated_at`, `deleted_at`（逻辑删除）
- 外键字段：以 `<table>_id` 命名，如 `box_id`
- 索引命名：`idx_<table>_<column>`

### 6.2 设计原则

1. **逻辑删除**：所有业务表必须包含 `deleted_at` 字段，采用逻辑删除。
2. **租户隔离**：所有业务表必须包含 `site_id` 字段，用于数据权限隔离。
3. **时序数据分离**：`state_stream` 作为高频写入表，单独设计，支持定期清理（默认30天）。
4. **报警数据长期保留**：`alarm_record` 保留90天，超期数据迁移至归档表。
5. **大字段分离**：图片URL存储在业务表，图片二进制存储在对象存储（MinIO/本地）。
6. **状态枚举使用SMALLINT**：数据库中以 `SMALLINT` 存储枚举值，Java中以枚举类映射。

### 6.3 核心表清单

| 表名 | 说明 | 生命周期策略 |
|------|------|-------------|
| `sys_user` | 系统用户 | 长期保留 |
| `sys_role` | 角色定义 | 长期保留 |
| `sys_user_role` | 用户角色关联 | 长期保留 |
| `sys_operation_log` | 操作日志 | 180天清理 |
| `site` | 站点信息 | 长期保留 |
| `edge_box` | 边缘盒子 | 长期保留 |
| `channel` | 视频通道 | 长期保留 |
| `state_stream` | 状态流原始数据 | 30天清理 |
| `pdi_task` | PDI作业记录 | 90天迁移 |
| `behavior_archive` | 行为档案 | 90天迁移 |
| `alarm_record` | 报警记录 | 90天迁移 |
| `archive_timeline` | 行为档案时间线 | 90天迁移 |
| `system_config` | 系统配置 | 长期保留 |
| `algorithm_config` | 算法配置 | 长期保留 |

## 9. 安全设计

### 9.1 认证机制

- **JWT Token**: 采用无状态JWT认证，Access Token有效期2小时，Refresh Token有效期7天
- **Token存储**: 客户端存储于HttpOnly Cookie或LocalStorage（由前端决定）
- **登出处理**: JWT加入Caffeine缓存黑名单，有效期与Token剩余有效期一致

### 9.2 数据权限隔离

- **站点隔离**: 所有业务查询自动拼接 `site_id = ?` 条件
- **数据权限过滤器**: 通过 Spring Data JPA 的 `Specification` 或自定义 `@DataScope` 注解实现
- **超级管理员**: `site_id` 条件自动跳过

### 9.3 密码安全

- **加密算法**: BCrypt（`BCryptPasswordEncoder`）
- **密码策略**: 最小8位，必须包含大写、小写、数字
- **登录锁定**: 连续5次失败锁定5分钟，失败次数存储于Redis/Caffeine

### 9.4 接口安全

- **HTTPS**: 生产环境强制HTTPS
- **CORS**: 配置白名单域名
- **XSS/CSRF**: 前后端分离模式下，通过JWT和CORS配置防护

## 10. 算法数据入口设计

### 10.1 设计目标

算法数据入口是连接边缘计算层与业务平台层的**专属边界模块**，要求：
1. **协议隔离**：将边缘盒子的私有数据协议转换为内部领域事件
2. **流量削峰**：状态流数据高频写入，需快速响应并异步处理
3. **扩展预留**：支持后续对接MQTT、Kafka等消息协议

### 10.2 入口层设计

```
┌─────────────────────────────────────────────────────────────┐
│                    Algorithm Inlet Module                   │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐ │
│  │ HTTP Receiver│  │ Data Parser │  │  Event Dispatcher   │ │
│  │ (Spring MVC) │  │ (JSON/Proto)│  │  (Spring Event)     │ │
│  └──────┬──────┘  └──────┬──────┘  └─────────────────────┘ │
│         │                │                                  │
│         └────────────────┘                                  │
│                          ↓                                  │
│         ┌─────────────────────────────────────┐             │
│         │         Validation Layer            │             │
│         │  • 盒子认证(Token/APIKey)           │             │
│         │  • 数据格式校验                     │             │
│         │  • 去重与幂等控制                   │             │
│         └─────────────────────────────────────┘             │
│                          ↓                                  │
│         ┌─────────────────────────────────────┐             │
│         │         Internal Event Bus          │             │
│         │  • StateStreamEvent                 │             │
│         │  • AlarmEvent                       │             │
│         │  • HeartbeatEvent                   │             │
│         └─────────────────────────────────────┘             │
└─────────────────────────────────────────────────────────────┘
```

### 10.3 核心接收协议

#### 状态流上报

### 10.4 异步处理机制

- 入口Controller接收数据后，立即将原始数据持久化到 `state_stream` 或 `alarm_record`
- 通过 `ApplicationEventPublisher` 发布内部领域事件
- `rule-engine` 模块监听事件，执行状态机转换与报警判定
- 入口接口响应时间目标 ≤ 50ms

## 12. 非功能性设计

### 12.1 性能设计

- **数据库连接池**: HikariCP，最大连接数 20
- **分页限制**: 单页最大 100 条
- **查询优化**: 所有列表查询必须命中索引，慢查询 (>1s) 触发告警
- **缓存策略**: 站点配置、用户信息、权限菜单使用 Caffeine 缓存，TTL 10分钟
- **实时推送**: SSE 替代 WebSocket，降低连接维护成本

### 12.2 可靠性设计

- **全局异常处理**: `@ControllerAdvice` 统一捕获并转换为标准响应
- **事务边界**: Service 层关键操作使用 `@Transactional`
- **数据校验**: `javax.validation` / `jakarta.validation` 在Controller层前置校验
- **降级策略**: 算法入口高负载时，优先保证接收接口可用，规则引擎异步处理可短暂排队

### 12.3 日志与监控

- **日志级别**: 生产环境默认 INFO，算法入口 DEBUG
- **日志格式**: JSON 结构化日志，含 traceId
- **健康检查**: `/actuator/health`
- **指标暴露**: `/actuator/metrics`
